# ウェイトリストLP 設計ドキュメント

- 日付: 2026-07-05
- ステータス: 承認済み
- 目的: 夫婦向け家計精算アプリ（仮称: **ヤマワケ**）のSaaS化需要を検証するため、ウェイトリスト登録LPを公開する

## 背景

現在のアプリは夫婦2人専用のプライベート運用。市場調査の結果、「収入合算→経費控除→余剰山分け」という精算モデルは既存アプリ（Splitwise / OsidOri / ワンバンク等）の空白地帯だが、市場規模は小さくマイクロSaaSが現実的上限と判断した。マルチテナント化に投資する前に、ウェイトリストLPで需要（登録数）と課金意向（価格提示への反応）を検証する。

## 決定事項

| 項目 | 決定 |
|------|------|
| 配置 | 既存リポジトリ内の `/lp` ルート（認証除外して公開） |
| 登録データ保存先 | 既存Cloudflare D1（Worker APIにエンドポイント追加） |
| 価格検証 | 想定価格を提示し、登録時に価格意向を選択させる |
| サービス仮称 | ヤマワケ |
| LP構成 | 精算シミュレーター付き1ページLP |

## 1. 配置とルーティング

- `src/app/lp/page.tsx` を新設。既存のNext.js 16 + Tailwind 4 + shadcn/ui をそのまま使う。
- `src/middleware.ts` のmatcherを変更し、`/lp` 配下を認証チェックの対象外にする。既存アプリの認証挙動（`/` や `/[year]/[month]` の保護）は変更しない。
- `/lp` 専用のメタデータ（title / description / OGP）を設定する。既存ページのメタデータは触らない。

## 2. ページ構成（1ページ、上から順）

1. **ヒーロー**: キャッチコピー「ふたりの収入から、ふたりの経費を引いて、残りを山分け。」+ サブコピー + 「ウェイトリストに登録」CTA（フォームセクションへスクロール）
2. **精算シミュレーター**: 夫婦それぞれの月収と共通経費の3つの数値入力 → 既存の `calculateSettlement`（`src/lib/utils/calculation.ts`、純関数）を再利用し「今月の精算額: ○○円（夫→妻）」を即時表示。クライアントコンポーネント。**入力値はサーバーに送信しない**ことをUI上に明記する。
3. **課題共感**: Excel精算・立替の記憶頼み・収入差の気まずさ、の3課題
4. **チーム思想（なぜ山分けか）**: 山分けモデルの本質的メリットを3つのシナリオで表現する。(a) 病気や事故でどちらかが働けなくなっても、同じ家計システムのまま協力して乗り越えられる、(b) 子育てで休んで給料が減っても「自分の給料が減る＝自分のお小遣いが減る」構図にならないから協力し合える、(c) 収入差があっても「ふたりで稼いでふたりで分ける」から対等。キーメッセージは「夫婦は常にチーム」
5. **機能紹介**: 月次記録 / 繰越管理 / 精算自動計算 の3点。実アプリのスクリーンショットは使わず、CSS/イラストによるモック表現とする
6. **価格 + 登録フォーム**: 想定価格「世帯 月380円（先行登録者は6ヶ月無料）」を明示。フォーム項目は (a) メールアドレス、(b) 価格意向ラジオ（`無料なら使いたい` = `free_only` / `月380円でも使いたい` = `paid_ok`）、(c) honeypotフィールド（非表示）。シミュレーターを一度でも操作したかはLP内のクライアント状態で保持し、送信時に `simulator_used` としてServer Actionへ渡す（シミュレーターの入力値自体は送らない）
7. **FAQ**: 3〜4問（いつ使える？ / データは安全？ / 夫婦以外でも使える？ 等）
8. **フッター**: 運営者表記・問い合わせ先

## 3. データ設計

### D1スキーマ（新規migration）

```sql
CREATE TABLE waitlist_entries (
  id TEXT PRIMARY KEY,
  email TEXT NOT NULL UNIQUE,
  price_intent TEXT NOT NULL CHECK (price_intent IN ('free_only', 'paid_ok')),
  simulator_used INTEGER NOT NULL DEFAULT 0,
  created_at TEXT NOT NULL
);
```

- `price_intent`: 課金許容層の比率を測る主要メトリクス
- `simulator_used`: シミュレーター操作の有無。体感が課金意向に効くかの分析用

### Worker API

- `POST /api/waitlist` を追加。**認証不要**（既存エンドポイントは全て認証必須のため、このルートのみ認証ミドルウェアをバイパスする）。
- 入力検証: email形式・price_intent列挙値。honeypotフィールドに値があれば、成功レスポンスを返しつつ保存しない。
- email重複（UNIQUE違反）は成功として返す（二重カウントせず、登録済みかどうかを外部から探れないようにする）。

### Next.js側

- Server Action `joinWaitlist`（`src/app/actions/waitlist.ts`）がZodバリデーション後にWorker APIを呼ぶ。既存の `src/lib/api/` クライアントのパターンを踏襲する。
- Zodスキーマは `src/lib/validations/` に配置（プロジェクト規約）。
- LPのフォーム・シミュレーターは `src/features/waitlist-lp/` に配置（Recursive Features Structure準拠、`index.tsx` のみ公開）。

## 4. エラーハンドリング

- バリデーションエラー: フォーム下にフィールド単位で日本語表示
- Worker API障害: 「時間をおいて再度お試しください」の汎用メッセージ（内部技術名は出さない）
- 成功: フォームを完了メッセージ（「登録ありがとうございます。準備ができたらご連絡します」）に置き換え

## 5. テスト

プロジェクトのTDD規約（RED→GREEN→REFACTOR、カバレッジ80%以上）に準拠。

- ユニット: waitlist Zodスキーマ、シミュレーター入力→ `calculateSettlement` への変換関数
- 統合: `joinWaitlist` Server Action（成功 / 重複 / バリデーションエラー / honeypot）
- コンポーネント: シミュレーターの入出力反映、フォームの送信中・完了状態
- E2E: `/lp` 表示 → シミュレーター操作 → 登録完了（MSW: `src/mocks/handlers.ts` にwaitlistハンドラー追加、インメモリDB `src/mocks/db.ts` にwaitlistテーブル追加）

## 6. デプロイ

- Next.js側: 既存のVercelデプロイに乗る（`/lp` が増えるだけ）
- Worker側: migration適用 + `wrangler deploy` が必要（Vercelプレビューだけ見てWorker本体のdeployを忘れない）

## 7. やらないこと（YAGNI）

- 外部アナリティクス（GA等）の導入 — 登録データ自体がD1にあり、検証の主要メトリクスはそれで足りる
- 確認メール送信・ダブルオプトイン
- A/Bテスト基盤、多言語対応、別ドメイン運用
- 実アプリ画面のスクリーンショット掲載（プライベートデータが写るため）

## 8. 成功基準（LP公開後の判断材料）

- ウェイトリスト登録 50〜100世帯: マルチテナント化の検討に値する
- `paid_ok` 比率 20%以上: 有料マイクロSaaSとして設計する根拠になる
- それ未満: 現状のプライベート運用を継続し、SaaS化は見送る
